# Adding & processing new content — the canonical runbook

**This file is the single source of truth for the content-ingestion
procedure.** If you add a pipeline step, add/rename a script, or change the
ordering, **update this file in the same commit** — an out-of-date runbook is
worse than none. `CLAUDE.md` points here; keep that pointer alive.

The procedure is meant to be **deterministic**: run the steps in order and you
get the same result every time. Most steps are pure functions of the mirror
(safe to re-run); the few stateful/agent-authored ones are marked.

---

## Two planes (read this first)

New content lands in one of two delivery planes, and they update differently:

- **R2 data plane** (reader/search/overview question panels, catalog, search
  index): fetched by the static pages **at runtime**. Refreshing it needs
  `publish --upload` / `build_search_index --upload` — **no site redeploy**.
- **HF audio plane** (`uild42/qb-audio`): the reader fetches `audio_index.json`
  + opus files **at runtime** (`cache: 'no-cache'` on the manifest). New audio
  is picked up on the next page load — **no site redeploy, no publish**. The
  uploader refreshes the manifest every 10 min.
- **Static site plane** (`output/**/*.html`, `index/wiki/reader/search.html`,
  shared JS): regenerated by `./build.sh` and shipped by the GitHub Pages
  deploy **on push to `main`**. New topic/overview/sweep **pages** need a push.

**So: new question text and new audio appear without a redeploy; new _pages_
need a build + push.**

## The ordered runbook for new sets

Steps are grouped; skip a group if it doesn't apply (noted per group). Do them
in this order — the ordering constraints below are load-bearing.

```bash
# 1. MIRROR SYNC — the only live-API step. Nothing sees new questions first.
qbmirror sync --db mirror/qbreader.sqlite       # --refresh "SET" for edits to an existing set

# 2. AUDIO (diff 7-9 tossups+bonuses) — see the Audio section for detail.
#    MSL queries its OWN copy of the mirror, so refresh it, then generate.
scp mirror/qbreader.sqlite msl:~/los_tts/qbreader.sqlite
ssh msl 'tmux new-session -d -s tts "~/venvs/chatterbox-tts/bin/python ~/los_tts/gen_tts.py > ~/los_tts/gen.log 2>&1"'
#    (uploader tmux `upload` runs upload_hf.py continuously; relaunch if dead)

# 3. OVERVIEWS / METADATA — per affected unit (lib/units.py). Agent steps noted.
python lib/sweep/author.py scaffold UNIT        # mechanical (surfaces NEW answerlines)
#   → agent edits sections.txt / intro.txt      (MANUAL — places new answerlines)
python lib/sweep/author.py assemble UNIT        # mechanical
python lib/sweep/capture_questions.py UNIT      # mechanical (needs step 1)
python lib/sweep/answerline_kb.py wikidata UNIT # mechanical / free
python lib/sweep/answerline_kb.py prep UNIT     # mechanical (incremental)
#   → agent enrichment batch_NN.out.jsonl       (MANUAL)
python lib/sweep/answerline_kb.py compose UNIT  # mechanical

# 4. SWEEP PAGE (optional — a per-tournament coverage page)
python lib/sweep/build_set.py "2024 New Tournament"   # --list-sets to find exact names

# 5. CROSS-REFS — only if NEW TOPIC PAGES were added (post_batch bundles these)
python lib/crossref/crossref.py                 # topic_index.json — MUST be first
python lib/crossref/relink.py                   # mutates analysis.json (local only)
python lib/crossref/infer.py                    # after relink; reads mirror

# 6. EMBEDDINGS + SEARCH — only for online semantic search over the new questions
python lib/embed/embed_corpus.py sentences      # resumable — skips already-embedded
python lib/embed/embed_corpus.py bonuses        # resumable
python lib/embed/build_search_index.py --upload # full rebuild + push to R2

# 7. PUBLISH the reader data plane to R2 (idempotent; ABORTS on dangling refs)
python lib/mirror/publish.py --upload

# 8. REBUILD static site + deploy (only if any HTML pages changed)
./build.sh
git add -A && git commit && git push origin main   # triggers the Pages deploy
```

**Minimal path** (just want the new questions playable/readable, no new pages):
`sync` → refresh MSL mirror + generate audio → `publish --upload`. Everything
else (overviews, crossref, search, site build) is additive and can follow.

## Audio in detail (`dev/tts/`, runs on MSL)

The generator queries a **copy** of the mirror on MSL, so new sets need the
copy refreshed before generating.

1. `scp mirror/qbreader.sqlite msl:~/los_tts/qbreader.sqlite` — refresh MSL's mirror.
2. Relaunch the generator: `ssh msl 'tmux new-session -d -s tts "~/venvs/chatterbox-tts/bin/python ~/los_tts/gen_tts.py > ~/los_tts/gen.log 2>&1"'`. It is **resumable/skip-existing**, so it only synthesizes questions whose `.opus` doesn't exist yet — i.e. exactly the new diff 7-9 tossups/bonuses.
3. Ensure the uploader is running: `ssh msl 'tmux ls'` should list `upload`. If not: `ssh msl 'cd ~/los_tts && tmux new-session -d -s upload "~/venvs/chatterbox-tts/bin/python upload_hf.py > upload.log 2>&1"'`. It commits new opus + a refreshed `audio_index.json` every 10 min.
4. Nothing else to do — the reader picks up the new manifest on the next load
   and the new questions enter audio-mode scope automatically.
- Monitor: `ssh msl 'grep made ~/los_tts/gen.log | tail -1'`.
- Scope/format/host decisions + the full setup live in `dev/tts/README.md`.
- To cover more difficulties, edit `DIFFS` in `dev/tts/gen_tts.py`.

## Hard ordering constraints (why the order is what it is)

- **sync (1) before everything** — nothing downstream sees new questions until
  the mirror has them. `publish` *aborts* with `DANGLING REF` if a committed
  ref (`questions_ref.json` / unit `questions.json`) points at an un-synced set.
- **refresh MSL mirror before audio generation** — MSL has its own copy.
- **overview `assemble` + answerline_kb `compose` before publish** — publish
  builds a **fresh** `SectionIndex`/`KBLookup` every run; answerlines not yet
  placed in an `overview.json` section or enriched in the KB publish as section
  `-1` ("Unsectioned") and stay unfaceted in the reader until authored.
- **capture_questions before publish** — publish resolves its `questions.json`
  refs into `unit_questions/`.
- **crossref.py before relink / infer / build / matcher** — all read
  `topic_index.json` (enforced in build.py and post_batch.py).
- **relink before infer** — infer excludes already-cross-ref'd targets.
- **embed_corpus before build_search_index** — the index only packs vectors
  already in `embeddings.sqlite`.
- **the set's text must be published before its sweep/overview panels show it**
  — panels resolve text from the R2 `sets/{slug}.json` shard at view time.

## Pure functions vs stateful

- **Pure functions of the mirror (+ committed overview/KB/ref files) — safe to
  always re-run, idempotent:** `publish.py`, `SectionIndex`, `capture_questions.py`,
  `answerline_kb` (wikidata/prep/compose skip existing), `build_set --rematch-only`,
  `crossref.py`, `build_search_index.py`, `embed_corpus.py` (resumable).
- **Stateful / one-way / agent-authored:** `sync` (network append), `relink.py`
  / `infer.py` (mutate analysis.json / write related.json), and the manual
  steps — `sections.txt`/`intro.txt`, KB `*.out.jsonl`, `crossref_overrides.json`,
  cards.

## Caveats

- **Question edits inside an already-mirrored set are NOT auto-detected.** Only
  `qbmirror sync --db mirror/qbreader.sqlite --refresh "SET NAME"` or a full re-seed
  (`qbmirror import-backup`) picks up edits; plain `sync` only adds
  wholly-missing sets.
- **New sets do NOT re-generate audio for old questions** — audio is keyed by
  question id and skip-existing, so a re-run only fills genuinely new questions.
- A reader/search page already open won't see new R2/HF data until reloaded
  (manifests load once per session).
